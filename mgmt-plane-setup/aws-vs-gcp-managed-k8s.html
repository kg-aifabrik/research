Recommendation

Prefer GCP (GKE Standard) over AWS (EKS) because: (1) GCP is ~$240/mo cheaper on-demand and ~$190/mo cheaper at 1-year commitments — driven entirely by networking: regional Cloud NAT saves $133/mo and the global HTTPS LB saves $98/mo; (2) GKE ships stronger K8s security defaults at no extra cost — Binary Authorization, Workload Identity, and GKE Security Posture are free and cover the key SOC 2 Type II supply-chain and threat-detection controls that on EKS require paid add-ons or manual integration; (3) GCP's global HTTPS LB uses a single anycast VIP, making multi-region active/passive failover a backend swap rather than a DNS record change. AWS advantage: larger ops-talent pool, broader third-party ecosystem, and more mature SOC 2 audit evidence — if your team is already AWS-fluent, that may outweigh the cost and security-defaults edge.

Requirements

  • Management plane governing 50+ remote sites; deployed in a single cloud (this report compares AWS vs GCP)
  • Managed K8s: EKS or GKE Standard on a private cluster (no public API endpoint)
  • Application workloads: 50 vCPU / 200 GB RAM per region
  • Managed PostgreSQL HA: RDS Multi-AZ or Cloud SQL HA
  • Multi-region active/passive DR: warm standby, RPO ≤ 5 min, RTO ≤ 30 min
  • Mixed exposure: public services (REST APIs + web console) fronted by Cloudflare; internal services via site-to-site IPsec VPN only
  • Cloudflare fronts all public ingress: WAF, TLS, DDoS, CDN
  • SOC 2 Type II: encryption at rest + in transit, audit logging, access controls
  • Staff SSO via Okta: kubectl OIDC, Grafana, web console
  • Self-hosted Prometheus + Grafana + Loki in-cluster; ~2.5 TB/mo log ingestion from 50 sites
  • Production only; costs at US list price (on-demand and 1-year committed)

Multi-region active/passive layout:

Primary (us-east-1 / us-central1)
  • EKS / GKE Standard · 3 AZs · 5 nodes
  • Full replicas — active traffic
  • RDS Multi-AZ / Cloud SQL HA
  • External LB + Internal LB
  • NAT · VPN Gateway · ArgoCD (primary)
  • WAL streaming → standby replica
Standby (us-west-2 / us-west1) — warm
  • EKS / GKE Standard · 3 AZs · 5 nodes
  • minReplicas: 1 — pre-warmed, idle
  • RDS Read Replica / Cloud SQL Replica
  • External LB + Internal LB
  • NAT · VPN Gateway · ArgoCD (watches same repo)
  • Promoted to primary on failover

ArgoCD in both regions watches the same Git repository. Standby workloads run at minReplicas: 1 to keep nodes warm. On failover: promote DB replica → scale up replicas via ArgoCD sync → update Cloudflare pool origin.

Global LB advantage: GCP's Cloud HTTPS LB uses a single global anycast VIP. During DR, switching from primary to standby is a backend service update — no DNS propagation needed. AWS ALBs are regional; each region has its own VIP and a Cloudflare DNS record or LB pool must be updated. Both are fast given Cloudflare controls DNS TTL, but GCP's path has fewer steps.
NAT Gateway cost note: AWS charges per Availability Zone — a standard 3-AZ region requires 3 NAT Gateways at $0.045/hr each. Two regions = 6 gateways = $197/mo fixed. GCP Cloud NAT is region-scoped; 1 gateway per region = $64/mo fixed. The $133/mo delta is purely topological.

Okta → kubectl Setup (Both Clouds)

  1. Create an Okta OIDC application; note the issuer URL and client ID.
  2. AWS: Register Okta as an EKS OIDC Identity Provider. Map Okta group claims to K8s ClusterRoleBinding. Users authenticate with kubectl oidc-login (kubelogin plugin).
  3. GCP: Configure GKE OIDC flags (--oidc-issuer-url, --oidc-username-claim) or use Anthos Identity Service. Users authenticate identically via kubelogin.

The integration complexity is equivalent on both clouds.

Disaster Recovery

StepActionTime
1. DetectCloudflare health monitor or external synthetic check fails on primary LB< 1 min
2. Promote DBPromote cross-region read replica to standalone primary (RDS promote-read-replica / Cloud SQL promote replica)~2–5 min
3. Scale workloadsTrigger ArgoCD sync with full replica counts in standby cluster, or kubectl scale directly~5–10 min
4. Cut trafficUpdate Cloudflare Load Balancing pool origin to standby LB VIP (or DNS record update)~1–2 min
5. ValidateAutomated smoke tests against standby API endpoint~5 min
Total RTO~15–25 min ✓
RPO detail: Async WAL replication typically lags < 30 s under steady-state load. Alert at replica_lag > 3 min to preserve the 5-min RPO buffer. Under heavy write bursts (bulk import, schema migration), lag can exceed 5 min — schedule maintenance windows accordingly. True RPO = 0 requires synchronous cross-region replication, which neither RDS nor Cloud SQL supports natively across regions (consider Aurora Global Database or AlloyDB for that requirement).
Automation note: Both AWS (Lambda + EventBridge) and GCP (Cloud Functions + Cloud Monitoring alerts) support automated DR runbook execution. Use a dead-man's-switch pattern: auto-promote only after confirmed primary unreachability; require human approval before the traffic cut to prevent split-brain.
